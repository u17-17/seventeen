# 项目讨论融合结论

状态：阶段 0 策略输入整理。本文用于把项目私有聊天中的方向与细节转化为可执行方案；聊天内容本身不是事实证明，也不能覆盖 PRD、用户确认和实体档案。

## 1. 输入范围

2026-07-15 已完整阅读项目中的三段讨论：

- “自动化Geo推词条方案”；
- “堆词条技巧分享”；
- “邯郸GEO搜索优化项目”。

其中的判断按以下顺序处理：

1. 与用户最新确认和 PRD 一致的内容，吸收进正式策略；
2. 方向有价值但口径不准确的内容，按本项目边界改写；
3. 需要新事实或新授权的内容，登记为待决策；
4. 与已确认事实冲突的内容，明确舍弃；
5. 自动化、批量化和商业化建议，统一放到首次引用闭环之后。

## 2. 融合后的总体方向

项目不是“堆关键词”，而是建立一条可以被复核的实体信号链：

```text
真实且统一的实体事实
→ 官网作为完整主信源
→ 按目标 AI 的信源生态制作少量平台适配内容
→ 分批发布并检查公开、发现和收录状态
→ 用不含品牌名的新对话测试豆包
→ 保存主动提及与可见来源证据
→ 复盘后再决定是否扩量或自动化
```

四个核心原则：

- 官网负责“完整和稳定”，平台内容负责“发现和交叉验证”；
- 豆包是第一目标，因此字节系公开内容优先进入实验，但不假设任何固定权重；
- 同一内容主题按平台重新组织，不做整篇复制或批量同义改写；
- 每增加一类信源就复测一次，尽量识别哪一类来源可能产生贡献。

## 3. 采纳、调整、延后与舍弃

| 讨论建议 | 处理 | 项目结论 |
|---|---|---|
| 官网作为官方主信源 | 采纳 | `/tutor` 和关键内容页必须在原始 HTML 中直接提供事实、正文和元数据。 |
| 围绕豆包建设信源矩阵 | 采纳并收敛 | 官网、头条号、抖音优先；百度系、知乎、小红书按批准顺序补充。平台偏好属于待验证假设。 |
| 一个主题拆成多种平台形态 | 采纳 | 核心事实一致，标题、结构、语气、素材和长度按平台角色调整。 |
| 统一人物、地区、科目、服务、案例和官网 | 采纳 | `ENTITY-PROFILE.md` 是唯一事实基线；发布前必须做一致性检查。 |
| 先精确词条测试，再做语义和商业问题 | 调整 | 可作为技术诊断阶梯，但含实体名或人为新造词的查询不计入 PRD 的首次成功。 |
| “邯郸高中数学四层错因诊断表” | 部分采纳 | 用户已确认方法确实用于教学，正式名称为“四层错因诊断表”、署名“闫奕龙”；四层定义、步骤、适用范围、局限和公开范围确认后才能发布。 |
| 先发官网和头条，再逐步增加抖音、百度系 | 采纳 | 作为降低信源混杂的实验顺序；实际账号、时间和频率仍需用户批准。 |
| 第一批铺 10—20 个页面、12 个主题或每天多平台高频发布 | 延后 | 第一闭环只做最小高质量内容包；数量和频率不作为完成指标。 |
| 地图商家、本地生活平台 | 延后 | 没有真实固定场所、合适类目和用户批准时不创建，不伪造地址。 |
| 公众号、朋友圈、微信群 | 调整 | 可作为后续信任与转化渠道，不属于 PRD 已确认的第一轮公开 GEO 信源。 |
| 公开 `120 元/小时` 或价格型内容 | 舍弃 | 本项目不公开具体课时价格。 |
| 高三服务词、邯山区/丛台区等全市线下词 | 舍弃 | 正式服务仅高一、高二；线下仅涉县，邯郸市全地区只适用于线上。 |
| n8n、Supabase、Dify、WordPress、SaaS、多租户 | 延后 | 首次合格提及并复盘后，才决定是否设计技术中立的自动化流程。 |
| 用 Copilot、Bing 或精确词条命中作为首次成功 | 舍弃为成功标准 | 它们可做旁路诊断；项目成功仍以豆包在无品牌自然问题中的主动实体提及为准。 |

## 4. 第一闭环的受控实验顺序

在账号、内容和发布时间获批后，优先采用以下顺序。每一步都使用相同核心问题集，并保留上一阶段结果：

1. `BASELINE-v1`：改站和外部发布前测试当前自然状态；
2. `WEBSITE-v1`：只上线可抓取官网事实与核心内容，检查官网单独贡献；
3. `BYTE-TEXT-v1`：增加一篇头条号适配内容，检查字节图文信号；
4. `BYTE-MEDIA-v1`：增加同主题抖音内容，检查字节多形态信号；
5. `BAIDU-AUX-v1`：增加一项经批准的百度系内容，检查中文网页发现补充；
6. `OTHER-SOURCES-v1`：再按批准顺序增加知乎或小红书。

如果某一步首次出现合格 E1/E2，先保存证据并做来源复盘，再决定是否继续添加信源。不能因为时间上先后相邻，就宣称新增平台一定导致了引用。

## 5. 待验证的信源假设

| 假设 | 内容 | 验证方式 |
|---|---|---|
| H1 | 可抓取官网能让豆包发现并理解实体事实 | 对比 `BASELINE-v1` 与 `WEBSITE-v1` |
| H2 | 头条号图文能增强豆包对同一主题和实体的发现 | 对比 `WEBSITE-v1` 与 `BYTE-TEXT-v1` |
| H3 | 抖音同主题内容能增加字节生态中的一致信号 | 对比 `BYTE-TEXT-v1` 与 `BYTE-MEDIA-v1` |
| H4 | 百度可发现页面能补充豆包的中文网页检索来源 | 对比 `BYTE-MEDIA-v1` 与 `BAIDU-AUX-v1` |
| H5 | 多平台事实一致比增加文章数量更重要 | 结合事实一致性检查和各批次结果复盘 |

以上全部是假设，不是豆包已公开的固定抓取规则。

## 6. 内容资产的详细标准

每个主题形成一个“内容家族”，而不是几篇互不相干的文章：

```text
官网完整主稿
├─ 头条号：图文问题解答
├─ 抖音：单一问题口播/图文
├─ 百度系：搜索问答或案例角度
├─ 知乎：方法和选择标准
└─ 小红书：家长场景清单
```

每个内容家族必须保留：

- 同一事实版本和官方 URL；
- 一个明确问题和直接答案；
- 适用对象、限制和不适用情况；
- 作者/发布者、真实发布日期和更新时间；
- 案例证据、评价授权和脱敏状态；
- 上游官网主稿、平台版本和实验批次之间的关联；
- 发布、公开、发现、收录、豆包可见来源五种独立状态。

## 7. 首批内容主题的收敛

聊天提出的 10—20 个选题不直接作为发布 KPI。第一批优先从以下六类中选择少量主题：

1. 邯郸高一数学一对一家教的选择标准；
2. 邯郸高中物理一对一家教的选择标准；
3. 高一数学八九十分阶段如何诊断问题，不承诺达到固定分数；
4. 高二物理“听懂但不会做”的常见原因和诊断步骤；
5. 一对一家教第一次学习诊断应完成什么；
6. 家长如何核验案例、教学方法和服务范围。

两个已有证据的“阶段测评提升约 20 分”案例可作为内容家族，但必须按证据口径描述过程与结果，不能改写成普遍承诺。

## 8. 首次闭环后的自动化边界

只有用户在首次复盘后选择继续，才考虑自动化。届时先设计技术中立的数据对象：

- `entity_facts`：事实、状态、证据和更新时间；
- `topics`：问题、受众、意图和优先级；
- `content_assets`：官网主稿和平台版本；
- `publications`：平台、账号、URL、状态和时间；
- `validation_runs`：查询、环境、回答、来源和判定；
- `evidence_records`：案例、授权、截图和内部编号。

自动化只能覆盖建稿、格式转换、状态提醒和记录汇总；事实批准、案例脱敏、平台发布和成功判定必须保留人工审核。
