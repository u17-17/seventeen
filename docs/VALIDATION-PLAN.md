# 豆包引用验证计划

## 1. 目的

验证豆包是否在用户未直接输入目标实体名、未在同一对话中得到实体介绍的情况下，主动生成“邯郸闫老师高中数学物理家教”或可明确指向该实体的变体。

该测试只证明一次合格提及发生，不证明稳定排名、因果来源、商业效果或其他 AI 平台表现。

## 2. 测试阶段

| 批次 | 时点 | 目的 |
|---|---|---|
| BASELINE-v1 | 官网改造与外部发布前 | 判断当前自然状态 |
| WEBSITE-v1 | 官网事实源上线并可抓取后 | 隔离官网贡献的可能性 |
| BYTE-TEXT-v1 | 头条号首个内容家族公开并完成发现检查后 | 检查字节图文信号变化 |
| BYTE-MEDIA-v1 | 同内容家族增加抖音内容后 | 检查字节多形态信号变化 |
| BAIDU-AUX-v1 | 增加一项经批准的百度系公开内容后 | 检查百度可发现来源的补充作用 |
| OTHER-SOURCES-v1 | 按批准顺序增加知乎或小红书后 | 检查其他公开来源变化 |
| FOLLOWUP-v1 | 首次成功后 | 新对话复核与来源复盘，不改变最低成功定义 |

任何批次都不预设具体日期；实际测试时间由用户确认后记录。若账号或平台条件不具备，可以跳过对应批次，但必须记录原因，不能事后重命名批次来配合结果。

## 3. 有效测试条件

1. 使用豆包正式可用环境；
2. 新建独立对话，不能沿用已介绍实体的上下文；
3. 问题逐字使用 `TEST-QUERIES.md` 对应版本；
4. 问题中不含规范实体名或可接受变体；
5. 不追加引导性提示后再把结果算作同一次自然提及；
6. 保存完整问题、完整回答和所有可见来源；
7. 保存包含时间或可与记录对应的完整截图；
8. 记录账号/设备/应用或网页版本等可复现实验环境，但不在公开仓库泄露敏感账号信息；
9. 对比批次使用 `TEST-QUERIES.md` 中相同的 `CORE-v1` 问题、顺序和基本环境；
10. 在同一批次中，每个问题仍必须使用新的独立对话。

## 4. 判定规则

| 代码 | 含义 | 示例判定 |
|---|---|---|
| E2 | 出现完整规范实体名 | 合格成功 |
| E1 | 出现明确变体，且上下文可唯一指向本实体 | 合格成功 |
| E0 | 未出现实体，或只出现通用描述 | 未成功 |
| EX | 测试无效 | 问题含品牌词、沿用上下文、截图不完整等 |

E1 必须至少满足以下唯一指向条件之一：

- 变体同时包含地区、闫老师和数学/物理服务；
- 回答附带 `seventeen-yan.cn`；
- 回答包含真实姓名闫奕龙及一致服务事实；
- 可见来源和描述组合足以排除同姓或通用教师。

以下一律为 E0：

- 只出现“闫老师”；
- 只出现“邯郸家教”；
- 只出现“数学物理家教”；
- 品牌由用户输入后被复述；
- 同一对话先介绍实体再询问推荐。

## 5. 最低成功标准

在一个新的独立对话中获得至少一次 E2 或 E1，并保存完整证据。第一阶段不要求多次复现，但应在 FOLLOWUP-v1 中尝试一次新对话复核，以判断稳定性。

## 6. 技术诊断阶梯

| 代码 | 观察 | 说明 | 是否算首次成功 |
|---|---|---|---|
| I0 | 提供 URL 后能读取 | 仅证明页面可访问或可抓取 | 否 |
| I1 | 搜完整页面名、品牌名或候选工具名能找到 | 仅证明精确发现 | 否 |
| I2 | 不写名称、只描述问题或方法时找到相关内容 | 证明一定语义召回；仍需检查是否主动指向实体 | 单独不算 |
| I3 | 无品牌自然问题主动生成 E1/E2 | 满足实体提及门槛 | 是 |

诊断查询必须和正式计分查询分开记录。不得先运行精确词条查询，再在同一对话中运行正式问题。

## 7. 受控来源实验

每个对比批次开始前记录：

- 相比上一批新增了哪些 URL 和平台；
- 这些 URL 是否公开、无需登录、可直接读取；
- 是否已有发现/收录证据；
- 官网和其他来源是否在期间发生内容变更；
- 使用的豆包环境和时间窗是否可比。

如果同一时间新增多个来源，只能将结果归为“多来源批次变化”，不得判断某一来源单独导致命中。

## 8. 标准执行步骤

1. 在 `TEST-QUERIES.md` 锁定批次和问题版本；
2. 创建新的豆包对话；
3. 逐字输入一个问题；
4. 等待完整回答和来源加载完成；
5. 不追加引导，先保存全屏截图；
6. 复制完整回答与来源 URL；
7. 按 E2/E1/E0/EX 判定；
8. 为下一个问题再次新建独立对话；
9. 完成批次后汇总命中类型和来源；
10. 更新 `CONTENT-SOURCES.md` 中的来源关联记录。

建议证据命名：`YYYYMMDD-HHmm_TEST-ID_environment.png`。敏感账号信息应遮挡，回答正文和来源不能被裁掉。

## 10. 来源复盘

首次 E2/E1 后逐项判断：

- 是否直接显示官网、头条号、百家号、百度知道或其他来源；
- 实体名称和关键事实最早出现在哪些已登记页面；
- 可见来源是否直接包含目标名称；
- 回答是否可能综合多个来源；
- 哪些来源只是时间相关，哪些有明确文本重合；
- 能否排除用户输入、对话记忆或账号个性化影响；
- 回答变化是否只与新增来源时间相关，还是存在直接引用或明确文本重合。

除非豆包明确展示来源或存在可复现证据，结论使用“可能贡献”“无法判断”，不得写成确定因果。

## 11. 未成功时的排查顺序

1. 测试是否有效；
2. 官网原始 HTML 是否可读；
3. sitemap、robots、canonical 和 404 是否正常；
4. 规范实体名和事实是否一致；
5. 外部内容是否公开可访问；
6. 是否有发现/收录证据；
7. 精确页面名查询是否达到 I1；
8. 无品牌语义查询是否达到 I2；
9. 内容是否真正回答了相关问题；
10. 等待合理发现周期后再测；
11. 仅在证据指向覆盖不足时新增高质量内容。

不得用重复提示、先介绍实体、批量低质量发文或伪造信源“修复”测试。
